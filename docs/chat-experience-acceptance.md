# Chat Experience 验收清单

## 目标

本清单用于验证聊天主链路在断线恢复、异常重试、流式反馈和长消息渲染上的一致性，确保用户能明确理解当前会话状态并继续工作。

## 验收项

- 当 WebSocket 首次建立连接时，页面应显示明确的连接状态提示，并在连接建立后自动消失。
- 当连接中断并进入重连时，页面应显示“正在恢复事件流”类提示，而不是静默卡住。
- 当恢复成功时，页面应显示已恢复事件数量或恢复完成提示。
- 当恢复失败时，页面应显示可理解的失败原因，并提供重试继续入口。
- 当用户中止当前回复时，页面应把本轮 assistant 消息标记为已中止，并允许立即重试。
- 当没有进行中请求却触发中止时，页面应返回“没有可中止请求”的明确信息，而不是无反馈。
- 当消息内容很长时，默认应折叠并提供展开/收起入口。
- 当消息中包含 Markdown 代码块时，代码块应有稳定样式与横向滚动能力，避免撑破布局。
- 工具调用卡片应至少展示名称、状态、摘要、耗时与展开后的输入/输出详情。
- 当消息或恢复流程失败时，错误提示应可读、可重试，不得只有原始异常字符串。

## 手工回归建议

1. 发起一次正常问答，确认 `message_started -> message_chunk -> message_done` 的状态流转正确。
2. 在流式输出中断网或刷新连接，确认页面进入恢复状态，并在恢复成功后继续展示最新消息。
3. 伪造 `qa.resume_failed`，确认页面出现错误提示与“重试继续”。
4. 在无进行中请求时发送 abort，确认页面展示“没有可中止请求”。
5. 发送一条超长 Markdown 消息，确认默认折叠且代码块样式稳定。
