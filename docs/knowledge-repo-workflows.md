# 知识与仓库工作流说明

当前知识链路已经形成从沉淀、注入、工作台编辑到团队共享的最小闭环。

## 1. 个人知识沉淀与工作台

- 个人知识通过知识建议卡或工作台直接创建，统一落在 `/knowledge` 工作台管理。
- 工作台支持关键词、分类、状态、标签筛选，并可通过 `/knowledge/:id` 深链接直接进入指定条目。
- 保存知识详情时会写入版本历史；共享状态变化也会同步写入知识版本，便于事后追溯。

## 2. 对话中的知识注入

- 仓库上下文构建分为两层：稳定背景知识和与当前问题相关的检索知识。
- 只有最终实际进入上下文的知识条目才会增加 `view_count`，候选但未注入的条目不计入使用统计。
- 回答完成后，聊天界面会展示本轮实际使用的知识引用，避免用户误判引用来源。

## 3. 团队共享流程

1. 作者在知识详情面板发起共享，请求会生成独立的共享记录，状态进入 `pending_review`。
2. 审核人可在知识共享审核页执行 `approve`、`request_changes`、`reject`、`publish`。
3. 只有状态为 `published` 的共享记录会进入团队知识库浏览结果。
4. 若作者根据审核意见重新提交，同一知识条目会复用原共享记录并刷新为新的待审状态，避免并行重复请求。

## 4. 审核约束

- 已归档知识不得发起共享。
- 同一知识在存在未终结共享请求时，不允许再次创建新的并行请求。
- `reject` 和 `request_changes` 必须填写审核备注，确保作者能看到明确处理意见。
- 团队知识浏览接口和页面只返回 `published` 条目，不暴露待审、驳回或退回修改中的共享记录。

## 5. 标准知识文件导入导出

- 单条导出使用 `Markdown 正文 + YAML frontmatter`，frontmatter 当前固定包含：
  `okk_format`、`format_version`、`title`、`summary`、`category`、`status`、`tags`、`source_refs`。
- 当前只支持 `format_version: 1`；导入未知版本、缺失必填字段或 frontmatter 非法时，会在预览阶段直接阻断，并返回按文件聚合的结构化错误。
- 批量导出会生成多份独立 `.md` 文件，并附带一个 `knowledge-export-manifest-*.json` 清单；适合落到仓库、压缩包或人工审阅目录中继续流转。
- 标准文件导入默认写入用户在导入页选择的目标仓库，不在协议里暴露内部知识 ID；协议聚焦稳定领域字段，而不是运行时瞬时字段。

## 6. 回滚与追溯策略

- 共享审核动作会追加独立审核记录，保存操作者、动作、备注和时间。
- 知识条目 `metadata.sharing` 会保留当前共享快照，用于个人工作台即时反馈。
- 标准文件导入在“预览”阶段不会写入知识库，只有执行“确认导入”后才会真正创建知识条目。
- 导入后的条目会在 `metadata.importBatchId` 与 `metadata.portability` 中保留批次号、格式版本和来源文件名；需要回滚时，应优先按导入批次定位，再结合知识版本历史执行删除或内容回退。
- 发布后的回滚依赖知识版本历史和共享审核轨迹联合判断；若需要撤销团队可见性，应先更新共享状态并再通过知识版本记录回溯内容变更。
- 排查争议时，优先同时查看知识版本历史和共享审核历史，确认内容变更与发布决策是否一致。

## 7. 知识订阅与更新消费

- 知识订阅面向消费侧，和共享审核分层：共享决定“哪些知识可见”，订阅决定“我持续跟踪哪些来源”。
- 当前订阅源统一抽象为三类：`team`、`project`、`topic`。其中 `project` 以仓库为来源，`topic` 以标签主题为来源。
- 订阅采用拉取式增量同步，不做实时推送；每条订阅都会记录最近同步时间、游标、同步状态与摘要。
- 首次打开未同步过的订阅时，系统会返回当前来源下可见的已发布共享知识作为初始更新集，并写入初始游标。
- 更新项消费状态至少区分 `pending`、`imported`、`duplicate`、`skipped`；已消费项应保留目标知识条目标识，便于跳转和排障。
- 一键导入仍复用既有知识创建语义；若目标仓库已存在同标题知识，则标记为 `duplicate`，而不是额外落一份新条目。

## 8. 订阅运营建议

- 团队级订阅适合关注全局发布动态；项目级订阅适合跟踪单仓库最佳实践；主题级订阅适合追踪如 `release`、`oncall`、`playbook` 这类跨仓库标签。
- 目标仓库应在创建订阅时明确指定，避免后续导入落点不一致，影响知识治理和回滚定位。
- 若某个订阅长期不再需要，应优先停用而不是删除，以保留同步轨迹和已消费更新的审计信息。
- 排查“为什么没收到更新”时，先检查订阅是否停用、最近同步是否成功、来源是否仍有 `published` 共享知识，以及游标是否已推进。
