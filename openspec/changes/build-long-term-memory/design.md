## Context

长期记忆决定了赛博合伙人能否真正表现出“持续理解用户和项目”的能力。当前系统只有相对静态的 Knowledge CRUD，缺少对用户偏好、项目事实、关系线索、过程经验和关键事件的结构化记忆建模，也缺少自动提取、检索和注入机制，导致每轮对话都要重新建立上下文。需要先把长期记忆从概念提升为正式系统能力，后续共享、治理和身份能力才有稳定基础。

当前 change 聚焦于长期记忆，需要把 proposal 中的能力边界落到可实现、可验证、可归档的设计基线。

## Goals / Non-Goals

**Goals:**
- 建立分类型记忆模型：明确偏好记忆、项目记忆、关系记忆、过程记忆和事件记忆的结构、生命周期和置信度
- 建立自动积累链路：从对话、执行步骤和用户反馈中提取可复用事实，并进行分类、去重和更新
- 建立记忆检索与注入策略：在新会话或关键步骤开始时，根据仓库、任务和用户上下文检索相关记忆并注入后端请求
- 建立记忆同步与互通：支持与 `CLAUDE.md` 等外部记忆形态双向同步，避免信息孤岛
- 建立衰减、刷新和人工管理机制：根据时效性、访问频率和用户确认对记忆进行刷新、降权或删除

**Non-Goals:**
- 不在本 change 中重写与当前主题无关的既有业务流程。
- 不为未来假设场景预留超出 proposal 范围的实现细节。
- 不绕过现有 capability 分层引入新的旁路实现。

## Decisions

1. 使用独立 capability 与 delta spec 表达 长期记忆 需求边界，避免实现先行导致范围漂移。
2. 涉及持久化时采用增量迁移与向后兼容策略，避免破坏既有数据与会话状态。
3. 所有关键状态变化都通过结构化接口或事件返回，便于前端消费、测试验证和后续治理。
4. 用户可见入口必须与后端能力同步落地，避免出现“底层可用但界面不可达”的割裂体验。
5. 验收以 capability 契约为准，测试、实现和后续归档均围绕同一份规格演进。

## Risks / Trade-offs

- 长期记忆 横跨 memory-system、ai-backend-engine、knowledge-engine、data-layer、web-backend、web-frontend 分层，若边界定义不清，后续实现容易出现职责重叠。
- 长期记忆 一旦缺少稳定状态模型或历史追踪，用户侧很难判断当前结果是否可信。
- 若实现阶段直接扩展超出 proposal 范围的功能，长期记忆 的交付周期和验证成本都会明显上升。

## Migration Plan

- 先落地数据模型、迁移脚本和索引调整，再接入上层服务逻辑。
- 随后实现核心运行时或服务层集成，确保 capability 契约先稳定。
- 最后交付 API 与前端入口，确保用户只看到已可用且可验证的能力。
- 完成后执行 OpenSpec 校验与针对该主题的实现级回归验证。

## Open Questions
