## Context

从 Personal Mode 走向 Collaboration Mode，核心不是把聊天窗口共享出去，而是让有价值的记忆和知识能够在团队中安全流动。当前记忆与知识默认是单用户视角，既无法支撑团队协作，也没有审核和权限边界来控制敏感信息传播。若要让 OKK 从“个人助手”进化成“团队合伙人”，就必须先把共享、审核和可见性模型正式化。

当前 change 聚焦于记忆共享，需要把 proposal 中的能力边界落到可实现、可验证、可归档的设计基线。

## Goals / Non-Goals

**Goals:**
- 建立记忆与知识可见性分级：支持 `private`、`workspace`、`shared` 等层级，并明确默认策略与切换规则
- 建立共享前审核流程：共享内容进入审核队列，由具备权限的成员确认、驳回或退回修改，降低敏感信息泄露风险
- 建立团队级推荐与复用：当成员处理相似任务时，优先推荐已经被审核通过的共享记忆和知识
- 建立团队知识仪表板：展示团队记忆积累、知识覆盖、贡献来源和待审核事项，便于持续运营
- **BREAKING** 调整底层数据模型：为记忆与知识增加可见性、审核状态和审批记录等字段

**Non-Goals:**
- 不在本 change 中重写与当前主题无关的既有业务流程。
- 不为未来假设场景预留超出 proposal 范围的实现细节。
- 不绕过现有 capability 分层引入新的旁路实现。

## Decisions

1. 使用独立 capability 与 delta spec 表达 记忆共享 需求边界，避免实现先行导致范围漂移。
2. 涉及持久化时采用增量迁移与向后兼容策略，避免破坏既有数据与会话状态。
3. 所有关键状态变化都通过结构化接口或事件返回，便于前端消费、测试验证和后续治理。
4. 用户可见入口必须与后端能力同步落地，避免出现“底层可用但界面不可达”的割裂体验。
5. 验收以 capability 契约为准，测试、实现和后续归档均围绕同一份规格演进。

## Risks / Trade-offs

- 记忆共享 横跨 memory-sharing、memory-system、knowledge-engine、data-layer、web-backend、web-frontend 分层，若边界定义不清，后续实现容易出现职责重叠。
- 记忆共享 一旦缺少稳定状态模型或历史追踪，用户侧很难判断当前结果是否可信。
- 若实现阶段直接扩展超出 proposal 范围的功能，记忆共享 的交付周期和验证成本都会明显上升。

## Migration Plan

- 先落地数据模型、迁移脚本和索引调整，再接入上层服务逻辑。
- 随后实现核心运行时或服务层集成，确保 capability 契约先稳定。
- 最后交付 API 与前端入口，确保用户只看到已可用且可验证的能力。
- 完成后执行 OpenSpec 校验与针对该主题的实现级回归验证。

## Open Questions

- 当前无阻塞性开放问题；若实现阶段引入新的外部依赖、权限边界或迁移风险，再回填对应约束。
