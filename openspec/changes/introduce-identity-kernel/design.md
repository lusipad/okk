## Context

赛博合伙人如果每次对话都只依赖当轮 prompt，就无法形成稳定的角色感、行为边界和用户适配能力。用户不仅需要一个“会回答问题的模型”，还需要一个能长期保持语气、偏好、协作方式和专业侧重点一致的合伙人。Identity Kernel 要解决的不是简单的 prompt 编辑，而是把“身份、风格、画像和注入方式”沉淀为可配置、可演进、可复用的正式能力。

当前 change 聚焦于身份内核，需要把 proposal 中的能力边界落到可实现、可验证、可归档的设计基线。

## Goals / Non-Goals

**Goals:**
- 建立身份配置模型：支持定义角色定位、语气风格、行为约束、专业偏好和禁用项，并保存多个版本
- 建立 System Prompt 编辑与切换能力：允许用户创建、预览、切换和回滚不同身份配置，适配不同工作场景
- 建立用户画像积累：从长期交互中提取用户技术栈、协作偏好和沟通习惯，作为身份的一部分持续优化输出
- 建立身份可见性与说明：提供 Partner Profile 视图，让用户明确看到当前身份配置、来源和最近变更
- 建立请求期注入机制：把身份配置和画像信息稳定注入后端请求，而不是散落在临时 prompt 中

**Non-Goals:**
- 不在本 change 中重写与当前主题无关的既有业务流程。
- 不为未来假设场景预留超出 proposal 范围的实现细节。
- 不绕过现有 capability 分层引入新的旁路实现。

## Decisions

1. 使用独立 capability 与 delta spec 表达 身份内核 需求边界，避免实现先行导致范围漂移。
2. 涉及持久化时采用增量迁移与向后兼容策略，避免破坏既有数据与会话状态。
3. 所有关键状态变化都通过结构化接口或事件返回，便于前端消费、测试验证和后续治理。
4. 用户可见入口必须与后端能力同步落地，避免出现“底层可用但界面不可达”的割裂体验。
5. 验收以 capability 契约为准，测试、实现和后续归档均围绕同一份规格演进。

## Risks / Trade-offs

- 身份内核 横跨 identity-kernel、ai-backend-engine、data-layer、web-backend、web-frontend 分层，若边界定义不清，后续实现容易出现职责重叠。
- 身份内核 一旦缺少稳定状态模型或历史追踪，用户侧很难判断当前结果是否可信。
- 若实现阶段直接扩展超出 proposal 范围的功能，身份内核 的交付周期和验证成本都会明显上升。

## Migration Plan

- 先落地数据模型、迁移脚本和索引调整，再接入上层服务逻辑。
- 随后实现核心运行时或服务层集成，确保 capability 契约先稳定。
- 最后交付 API 与前端入口，确保用户只看到已可用且可验证的能力。
- 完成后执行 OpenSpec 校验与针对该主题的实现级回归验证。
