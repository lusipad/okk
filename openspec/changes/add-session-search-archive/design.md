## Context

会话历史是赛博合伙人最直接、最容易被忽略的长期资产。当前会话列表主要依赖时间排序，缺少搜索、归档和引用能力，导致过去的讨论、结论和排障过程很快沉没，用户不得不反复描述同一背景。把会话从“用完即走的聊天记录”升级为“可搜索、可沉淀、可复用的工作档案”，是提升长期使用价值的必要一步。

当前 change 聚焦于会话搜索与归档，需要把 proposal 中的能力边界落到可实现、可验证、可归档的设计基线。

## Goals / Non-Goals

**Goals:**
- 建立会话全文检索：对标题、消息正文和会话摘要建立检索索引，支持关键字搜索、命中高亮和结果排序
- 建立归档与恢复机制：支持用户主动归档不活跃会话，并在需要时恢复，保持主列表聚焦当前工作
- 建立标签和筛选能力：为会话增加标签、状态和筛选条件，方便按主题或阶段组织历史内容
- 建立跨会话引用：在当前会话中引用历史对话片段、摘要或结论，减少重复描述和上下文搬运
- 建立摘要辅助检索：在会话结束或空闲时生成摘要，提升搜索召回质量和归档后的可读性

**Non-Goals:**
- 不在本 change 中重写与当前主题无关的既有业务流程。
- 不为未来假设场景预留超出 proposal 范围的实现细节。
- 不绕过现有 capability 分层引入新的旁路实现。

## Decisions

1. 使用独立 capability 与 delta spec 表达 会话搜索与归档 需求边界，避免实现先行导致范围漂移。
2. 涉及持久化时采用增量迁移与向后兼容策略，避免破坏既有数据与会话状态。
3. 所有关键状态变化都通过结构化接口或事件返回，便于前端消费、测试验证和后续治理。
4. 用户可见入口必须与后端能力同步落地，避免出现“底层可用但界面不可达”的割裂体验。
5. 验收以 capability 契约为准，测试、实现和后续归档均围绕同一份规格演进。

## Risks / Trade-offs

- 会话搜索与归档 横跨 data-layer、web-backend、web-frontend 分层，若边界定义不清，后续实现容易出现职责重叠。
- 会话搜索与归档 一旦缺少稳定状态模型或历史追踪，用户侧很难判断当前结果是否可信。
- 若实现阶段直接扩展超出 proposal 范围的功能，会话搜索与归档 的交付周期和验证成本都会明显上升。

## Migration Plan

- 先落地数据模型、迁移脚本和索引调整，再接入上层服务逻辑。
- 最后交付 API 与前端入口，确保用户只看到已可用且可验证的能力。
- 完成后执行 OpenSpec 校验与针对该主题的实现级回归验证。

## Open Questions

- 当前无阻塞性开放问题；若实现阶段引入新的外部依赖、权限边界或迁移风险，再回填对应约束。
