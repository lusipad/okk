## Context

知识会随着代码演进、外部依赖变化和团队决策更新而逐步失真。当前系统虽然已经有知识状态和版本记录，但还没有把“过时检测、来源追踪、冲突处理、治理闭环”提升为正式能力，这会直接影响知识可信度，也会拖累后续长期记忆、共享知识和自动推荐能力的质量。需要先建立治理底座，才能让知识积累真正可持续。

当前 change 聚焦于知识治理，需要把 proposal 中的能力边界落到可实现、可验证、可归档的设计基线。

## Goals / Non-Goals

**Goals:**
- 建立知识过时检测：结合时间衰减、仓库变更、引用失效和人工反馈，主动标记可能失真的知识条目
- 建立来源追踪：记录知识从哪次会话、哪个仓库、哪个 Agent、哪条操作链路中产生，并保留原始证据引用
- 建立冲突发现与处理流程：识别语义矛盾、重复覆盖或版本分叉的知识，并进入待审核队列
- 建立知识版本比对：支持查看知识演进历史、变更摘要和前后差异，方便审核和回滚
- 建立治理工作台：集中展示健康度、过时清单、冲突清单、待审核事项和治理操作入口

**Non-Goals:**
- 不在本 change 中重写与当前主题无关的既有业务流程。
- 不为未来假设场景预留超出 proposal 范围的实现细节。
- 不绕过现有 capability 分层引入新的旁路实现。

## Decisions

1. 使用独立 capability 与 delta spec 表达 知识治理 需求边界，避免实现先行导致范围漂移。
2. 涉及持久化时采用增量迁移与向后兼容策略，避免破坏既有数据与会话状态。
3. 所有关键状态变化都通过结构化接口或事件返回，便于前端消费、测试验证和后续治理。
4. 用户可见入口必须与后端能力同步落地，避免出现“底层可用但界面不可达”的割裂体验。
5. 验收以 capability 契约为准，测试、实现和后续归档均围绕同一份规格演进。

## Risks / Trade-offs

- 知识治理 横跨 knowledge-governance、knowledge-engine、data-layer、web-backend、web-frontend 分层，若边界定义不清，后续实现容易出现职责重叠。
- 知识治理 一旦缺少稳定状态模型或历史追踪，用户侧很难判断当前结果是否可信。
- 若实现阶段直接扩展超出 proposal 范围的功能，知识治理 的交付周期和验证成本都会明显上升。

## Migration Plan

- 先落地数据模型、迁移脚本和索引调整，再接入上层服务逻辑。
- 随后实现核心运行时或服务层集成，确保 capability 契约先稳定。
- 最后交付 API 与前端入口，确保用户只看到已可用且可验证的能力。
- 完成后执行 OpenSpec 校验与针对该主题的实现级回归验证。
