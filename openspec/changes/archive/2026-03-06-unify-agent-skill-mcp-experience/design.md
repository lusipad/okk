## Context

整体架构已经把 Skill / MCP / 多 Agent 定义为统一协作能力，但当前实现更偏“能力分别存在”：Agent 有自己的事件、MCP 有自己的配置页、Skill 有自己的安装闭环，用户在主对话里无法低成本理解它们之间的关系。

## Goals / Non-Goals

**Goals:**
- 为 Agent、Skill、MCP、Backend Tool 建立统一运行时语言
- 让主工作台能直接看到能力是否可用、谁在执行、哪里失败
- 让失败恢复具备明确动作，而不是只暴露错误文本
- 将协作能力收敛为一套可复用的前后端契约

**Non-Goals:**
- 不在本 change 中做视觉系统细节打磨
- 不处理知识条目生命周期与仓库上下文设计
- 不重写 Desktop 包装层的全部行为

## Decisions

1. 所有协作运行单元都需要统一的 run id、source type、status 和 diagnostics 字段。
2. 主对话区域只显示摘要，详细运行信息进入统一协作侧栏和时间线。
3. 能力可用性必须前置暴露，例如 backend 缺失、MCP 未启用、Skill 安装失败，不允许等到执行时才模糊报错。
4. 失败恢复至少提供重试、查看详情、跳转到配置页三种动作之一。
5. WebSocket 实时事件与 REST 查询结果必须共享同一结构化语义，避免前端双重解析。

## Risks / Trade-offs

- 统一事件模型会牵动多个模块，若边界不清容易演变成“一次性重写”。
- 前端展示过度详细会干扰主对话，因此需要遵守默认折叠、异常放大的原则。
- 后端若不能保证幂等 run id 和稳定状态迁移，前端时间线会出现重复或错序。
