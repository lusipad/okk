## Context

知识条目的 CRUD、搜索、版本历史和状态更新后端能力已经存在，前端当前只有治理页、导入页和对话侧栏中的零散入口，没有统一的知识工作台。这个 change 只补前端可达性与主工作流，不改动知识数据模型和后端接口语义。

## Goals / Non-Goals

**Goals:**
- 交付稳定的知识主页面，覆盖搜索、筛选、列表、详情和编辑主链路
- 在现有前端 IOProvider 中补齐知识 CRUD、搜索、版本历史和状态更新桥接
- 提供稳定的 `/knowledge` 与 `/knowledge/:id` 深链接，支持从列表和外部入口直达条目
- 将 Knowledge 提升为一级导航入口，让知识工作台成为日常主任务

**Non-Goals:**
- 不在本 change 中改动 `/api/knowledge` 的 REST 语义或新增后端端点
- 不在本 change 中实现 `KnowledgeSuggestionCard` 的保存前编辑与保存后跳转
- 不在本 change 中重写知识排序算法、质量评分或治理规则

## Decisions

1. 采用单页双区结构承载知识工作台：左侧负责搜索、筛选和列表，右侧负责详情、编辑和版本历史。
   - 备选方案：拆成列表页和详情页。拒绝原因：会增加路由切换与状态同步复杂度，不适合高频检索和快速编辑。
2. 以 URL 作为主定位信息，`/knowledge/:id` 只表达当前聚焦条目，筛选条件继续保留在页面状态中。
   - 备选方案：把全部筛选条件都写入 URL。拒绝原因：当前 change 重点是可达性与主工作流，不需要提前引入复杂的查询串状态管理。
3. 前端直接复用既有知识 REST 接口，不新增聚合 backend。
   - 备选方案：增加专用 KnowledgePage 聚合接口。拒绝原因：现有接口已覆盖所需能力，新增聚合层只会扩大变更面。
4. 知识详情的保存、状态切换和删除完成后统一回刷列表与详情数据，避免在本 change 中引入额外缓存层。
   - 备选方案：为知识列表建立全局 optimistic cache。拒绝原因：当前没有足够证据证明需要复杂缓存，直接回刷更稳妥。
5. 版本历史只提供查看与切换浏览，不在本 change 中加入复杂的 diff 可视化。
   - 备选方案：在工作台内直接实现富文本 diff。拒绝原因：会显著扩大前端复杂度，且 proposal 未要求。

## Risks / Trade-offs

- [页面状态较多] → 用“列表筛选状态”和“当前条目状态”两组职责拆分，避免所有状态堆在单组件中
- [列表与详情回刷造成额外请求] → 维持接口简单性，先保证正确性；后续若确有性能问题，再独立做缓存优化 change
- [一级导航调整影响整体工作台密度] → 只调整 Knowledge 入口层级，不连带改动其他导航分组

## Migration Plan

- 先补齐 IOProvider 的知识桥接方法与页面数据装配
- 再交付 `KnowledgePage`、详情编辑面板和版本历史视图
- 最后接入路由与一级导航，并补齐页面级测试
- 如需回滚，移除 `/knowledge` 路由与导航入口即可，后端和数据库无需回退

## Open Questions

- 当前无阻塞性开放问题；若实现时发现版本历史展示需要更复杂 diff，再通过独立 change 处理
