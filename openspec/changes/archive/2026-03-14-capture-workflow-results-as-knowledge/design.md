## Context

当前工作流系统能够保存运行输入、输出和步骤状态，但运行结束后结果仍停留在 `skill_workflow_runs` 记录中，无法自然转化为知识条目。与此同时，知识系统已经具备 CRUD、版本历史、状态流转和知识建议保存链路，因此工作流结果沉淀没有必要另起一套知识模型，关键是定义稳定的映射规则、发布入口和模板预置方式。

这个 change 聚焦于“工作流结果发布为知识”，而不是知识作为输入。设计要复用现有知识写入能力，同时保证运行详情中能显式确认、编辑和回流到知识条目。

## Goals / Non-Goals

**Goals:**
- 为工作流运行记录提供“保存为知识”入口，支持保存完整结果或结果摘要。
- 定义工作流运行结果到知识条目的默认映射，包括标题、摘要、正文、标签和来源元数据。
- 为代码审查沉淀、上下文整理、知识健康检查提供预置模板元数据，使模板天然支持知识发布。
- 在前端运行详情中补齐发布前确认、编辑和发布成功后的回流能力。

**Non-Goals:**
- 不在本 change 中扩展新的工作流输入节点类型；该能力由 `extend-workflows-with-knowledge-nodes` 负责。
- 不在本 change 中引入新的知识状态机；发布后的条目仍复用现有 `draft/published/stale/archived` 流程。
- 不在本 change 中实现团队审核、共享和订阅；这些能力由知识共享相关 change 负责。

## Decisions

1. **复用现有知识创建链路，而不是新增独立知识表**
   - 选择：工作流发布动作最终写入现有 `KnowledgeEntry`，并在 `metadata` 中附带 `workflowId`、`runId`、`templateId`、`sourceStepIds`。
   - 原因：避免为工作流结果另建平行知识模型，保证知识治理、搜索和版本管理都能直接复用。
   - 备选：新增 `workflow_knowledge_entries` 独立表。
   - 放弃原因：会把“工作流产物”与“普通知识条目”割裂，增加治理复杂度。

2. **发布入口挂在工作流运行记录之上**
   - 选择：新增基于运行记录的发布动作，例如 `POST /api/workflows/runs/:runId/publish-knowledge`，由后端从运行记录生成默认知识负载。
   - 原因：运行记录已经包含输入、输出和步骤详情，是最稳定的发布源。
   - 备选：由前端直接调用现有 `/api/knowledge` 并自行拼装 payload。
   - 放弃原因：会把映射规则散落到前端，导致模板和运行详情难以保持一致。

3. **模板只提供默认映射，不强制自动发布**
   - 选择：模板中增加与知识沉淀相关的默认元数据，例如建议标题前缀、默认分类、标签和推荐保存模式，但是否落库始终由用户显式触发。
   - 原因：保留用户对知识质量的最终控制权，避免每次运行都制造低质量草稿。
   - 备选：模板运行完成后自动生成知识。
   - 放弃原因：会放大噪音和重复条目，增加治理成本。

4. **前端发布前允许编辑默认映射结果**
   - 选择：运行详情打开保存面板时，前端展示后端生成的默认标题、摘要、正文、标签和分类，用户可在保存前调整。
   - 原因：运行输出往往偏机器化，直接保存会降低知识可读性。
   - 备选：只提供“一键保存”。
   - 放弃原因：难以保证知识质量，也和现有知识建议保存链路不一致。

## Risks / Trade-offs

- [运行输出过长] → 提供“完整结果”和“摘要结果”两种保存模式，并由后端生成默认摘要。
- [模板元数据与实际输出不匹配] → 发布接口始终以运行记录为准生成默认负载，模板只作为初始偏好。
- [发布动作与现有知识建议链路重复] → 统一复用知识创建语义与前端编辑交互，不另造第二套知识保存体验。

## Migration Plan

- 先定义工作流结果到知识条目的映射模型与模板元数据结构。
- 再补齐工作流运行记录上的发布接口和知识写入逻辑。
- 最后实现前端运行详情中的保存面板、模板展示和成功回流。

## Open Questions

- 当前默认保存为 `draft` 知识条目；如后续需要模板级“直接 published”策略，应另开 change 明确审核边界。
- 当前不支持把单个运行拆成多条知识批量保存，先覆盖单次运行生成单条知识的主链路。
