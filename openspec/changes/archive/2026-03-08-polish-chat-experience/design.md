## Context

聊天是用户感知 OKK 能力的第一触点，稳定性和细节质量直接决定“这个合伙人靠不靠谱”。当前聊天链路在流式状态管理、工具调用展示、错误恢复和长消息处理上仍然存在明显摩擦，导致即使底层能力已经具备，用户也会因为体验断裂而怀疑系统可靠性。Phase 1.5 需要先把聊天体验从“能用”提升到“可信、顺手、可恢复”。

当前 change 聚焦于聊天体验打磨，需要把 proposal 中的能力边界落到可实现、可验证、可归档的设计基线。

## Goals / Non-Goals

**Goals:**
- 加固流式会话状态机：处理 WebSocket 断连重连、事件丢失恢复、背压控制和消息顺序一致性
- 重构工具调用呈现：以更清晰的卡片展示工具名称、输入输出摘要、状态、耗时和可展开详情
- 建立错误恢复与重试：为超时、CLI 异常、恢复失败和用户中止提供明确状态与重试动作
- 优化消息渲染：提升 Markdown、代码块、高亮、长消息分段加载和空态反馈体验
- 建立会话级中断与恢复能力：允许用户在执行中中止当前链路，并在需要时继续同一上下文

**Non-Goals:**
- 不在本 change 中重写与当前主题无关的既有业务流程。
- 不为未来假设场景预留超出 proposal 范围的实现细节。
- 不绕过现有 capability 分层引入新的旁路实现。

## Decisions

1. 使用独立 capability 与 delta spec 表达 聊天体验打磨 需求边界，避免实现先行导致范围漂移。
2. 优先复用现有分层，不为 聊天体验打磨 引入不必要的新基础设施。
3. 所有关键状态变化都通过结构化接口或事件返回，便于前端消费、测试验证和后续治理。
4. 用户可见入口必须与后端能力同步落地，避免出现“底层可用但界面不可达”的割裂体验。
5. 验收以 capability 契约为准，测试、实现和后续归档均围绕同一份规格演进。

## Risks / Trade-offs

- 聊天体验打磨 横跨 web-frontend、web-backend、ai-backend-engine 分层，若边界定义不清，后续实现容易出现职责重叠。
- 聊天体验打磨 一旦缺少稳定状态模型或历史追踪，用户侧很难判断当前结果是否可信。
- 若实现阶段直接扩展超出 proposal 范围的功能，聊天体验打磨 的交付周期和验证成本都会明显上升。

## Migration Plan

- 随后实现核心运行时或服务层集成，确保 capability 契约先稳定。
- 最后交付 API 与前端入口，确保用户只看到已可用且可验证的能力。
- 完成后执行 OpenSpec 校验与针对该主题的实现级回归验证。

## Open Questions
