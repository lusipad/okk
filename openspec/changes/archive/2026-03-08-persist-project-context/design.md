## Context

用户在不同项目中的工作方式往往并不相同：有的仓库偏向快速原型，有的仓库强调严格测试和发布流程，还有的仓库依赖特定 Agent、Skill 和约定。当前 RepositoryContextService 主要读取 `CLAUDE.md` 和知识摘要，无法持续记住用户在某个仓库里的偏好、最近操作和常见模式，导致“继续上次工作”只能依赖用户重复说明。需要把项目级上下文持久化为正式能力，让合伙人在进入仓库时就带着正确背景开始协作。

当前 change 聚焦于项目上下文持久化，需要把 proposal 中的能力边界落到可实现、可验证、可归档的设计基线。

## Goals / Non-Goals

**Goals:**
- 建立仓库级偏好模型：记录默认 Agent、常用 Skill、偏好的工作模式和关键项目约定
- 建立工作模式识别：从会话和执行记录中归纳用户在该仓库中的常见任务模式，如调试、重构、评审和发布
- 建立上下文自动注入：在新会话开始时根据仓库自动加载项目偏好、最近范围和相关历史操作
- 建立最近操作记忆：保存最近一次任务的关键步骤、结果和待续事项，支持“继续上次工作”
- 建立用户可见的项目上下文入口：让用户可以查看、修正或清理仓库级偏好和工作模式

**Non-Goals:**
- 不在本 change 中重写与当前主题无关的既有业务流程。
- 不为未来假设场景预留超出 proposal 范围的实现细节。
- 不绕过现有 capability 分层引入新的旁路实现。

## Decisions

1. 使用独立 capability 与 delta spec 表达 项目上下文持久化 需求边界，避免实现先行导致范围漂移。
2. 涉及持久化时采用增量迁移与向后兼容策略，避免破坏既有数据与会话状态。
3. 所有关键状态变化都通过结构化接口或事件返回，便于前端消费、测试验证和后续治理。
4. 用户可见入口必须与后端能力同步落地，避免出现“底层可用但界面不可达”的割裂体验。
5. 验收以 capability 契约为准，测试、实现和后续归档均围绕同一份规格演进。

## Risks / Trade-offs

- 项目上下文持久化 横跨 repo-management、data-layer、ai-backend-engine、web-frontend 分层，若边界定义不清，后续实现容易出现职责重叠。
- 项目上下文持久化 一旦缺少稳定状态模型或历史追踪，用户侧很难判断当前结果是否可信。
- 若实现阶段直接扩展超出 proposal 范围的功能，项目上下文持久化 的交付周期和验证成本都会明显上升。

## Migration Plan

- 先落地数据模型、迁移脚本和索引调整，再接入上层服务逻辑。
- 随后实现核心运行时或服务层集成，确保 capability 契约先稳定。
- 最后交付 API 与前端入口，确保用户只看到已可用且可验证的能力。
- 完成后执行 OpenSpec 校验与针对该主题的实现级回归验证。

## Open Questions
