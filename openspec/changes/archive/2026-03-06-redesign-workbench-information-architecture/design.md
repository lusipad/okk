## Context

整体架构已经把“统一工作台”定义为核心产品形态，但现有实现仍带有原型阶段痕迹：一级导航、会话历史、能力入口、上下文信息没有形成稳定层级，导致用户切换 Chat / Skills / MCP 时缺少一套可预测的心理模型。

## Goals / Non-Goals

**Goals:**
- 把产品从“若干页面”收敛成“一个工作台”
- 明确左中右三块区域各自承载的职责
- 让命令面板、专注模式、响应式抽屉成为信息架构的一部分
- 确保 Desktop 与 Web 使用同一套工作台结构

**Non-Goals:**
- 不在本 change 中定义完整视觉风格细节
- 不重写 Agent / Skill / MCP 的运行时逻辑
- 不处理发布、打包或 CI 门禁细节

## Responsibility Audit

| 区域 | 变更前冲突 | 本次收口 |
| --- | --- | --- |
| 左侧导航 | 一级能力入口、会话历史和搜索混在一起，用户无法判断“切页面”与“切会话”的差别 | 左栏固定分成 `New chat / Search / Primary links / Chats` 四层；一级入口只负责进入点，会话历史独立承载回访 |
| 命令入口 | 顶栏按钮、快捷键、Desktop 搜索入口各自为政，行为容易分叉 | `ShellLayout` 统一收口为命令面板入口；快捷键、顶栏按钮、桌面 search bridge 和自定义事件全部走同一打开逻辑 |
| 右侧上下文 | Team、事件、知识建议和诊断信息长期暴露，阅读噪音高 | 右栏改为默认按需展开，并根据运行态自动推荐 `overview / timeline / knowledge`，同时允许用户显式固定 tab |
| 专注模式 | 左右栏、抽屉、上下文恢复规则没有统一约束 | 专注模式只保留主舞台，左右栏统一隐藏；刷新后恢复偏好；移动端抽屉与桌面栏位共用同一状态语义 |
| Desktop 壳层 | 桌面全局搜索/唤起入口可能绕开当前工作台语义 | preload 暴露统一 `search.onQuery` 订阅，桌面查询直接回到当前工作台并打开同一个命令面板 |

## Decisions

1. 左侧只承载“导航与进入点”，不承载复杂运行信息。
2. 中间区域始终是主任务舞台，Chat、知识编辑、设置详情都在这里完成。
3. 右侧面板统一承载上下文信息，例如步骤、知识建议、Team 状态、诊断信息，并采用按需展开。
4. 命令面板不是附加小工具，而是一级导航补充层，负责跳转、搜索、模式切换。
5. Desktop 必须复用同一信息架构，不能出现“Web 一套、桌面另一套”的导航逻辑。

## Navigation And State Ownership

| 层级 | 入口 / 语义 | 路由 / 展示位置 | 状态归属 | 持久化 / 恢复规则 |
| --- | --- | --- | --- | --- |
| `New chat` | 创建新会话并回到主舞台 | 左栏按钮，主舞台仍为 `/` | Chat store 管理当前会话与消息 | 复用现有会话创建逻辑，不额外持久化 |
| `Search` | 工作台级搜索与命令跳转 | 左栏 Search 分区 + 命令面板 | `ShellLayout` 管理命令面板开关与查询词 | 支持快捷键、桌面 query、自定义事件统一恢复 |
| `Primary links` | 顶层能力页面切换 | `/`、`/skills`、`/settings/mcp` | Router 管理页面，页面级 store 管理内容 | 返回时保持最近的会话、仓库与右栏状态 |
| `Chats` | 历史会话回访 | 左栏历史列表 | Chat store 管理当前会话 id | 会话选择不应改写一级导航选择 |
| 右侧上下文 | 概览、时间线、任务图、知识建议 | 右栏面板 | `RightSidebar` 管理 tab、过滤与局部交互 | 记住用户显式 tab 偏好；无偏好时按运行态自动选默认 tab |
| 专注模式 | 只保留主舞台，屏蔽两侧干扰 | `ShellLayout` 栅格 / 抽屉 | `ShellLayout` 管理 focusMode | `localStorage` 持久化 `okk.focus-mode` |
| 右栏展开态 | 桌面协作面板或移动端上下文抽屉 | `ShellLayout` 右栏 / 抽屉 | `ShellLayout` 管理 rightOpen | `localStorage` 持久化 `okk.right-panel-open` |
| Desktop 搜索桥 | 外部搜索把用户带回当前工作台 | Electron preload -> Web Shell | preload 负责桥接，`ShellLayout` 负责呈现 | 订阅 `search.onQuery` 时自动清理，避免重复监听 |

## Responsive Rules

- Desktop 默认展示左栏，右栏按需展开；命令面板为覆盖层，不改变当前路由。
- Mobile 通过顶部按钮打开左右抽屉；抽屉只是工作台的响应式表现，不是独立页面流。
- 进入专注模式后，左右栏和抽屉统一关闭，只保留主舞台与命令入口。
- 桌面与 Web 对命令面板、右栏恢复、专注模式使用相同语义，避免跨端记忆割裂。

## Risks / Trade-offs

- 若 change 边界控制不好，容易与视觉系统 change 互相穿插。
- 路由与布局状态重构会影响现有测试，需要同步更新验收用例。
- 如果右栏承载过多内容，仍可能回到“信息噪音过高”的旧问题，因此必须坚持默认收起、异常放大原则。
