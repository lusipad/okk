## Context

Skill 是合伙人“成长”支柱的核心载体，但当前 Skill 系统仍停留在“扫描目录 + 解析 `SKILL.md`”的初级阶段，既缺少渐进式加载和版本状态管理，也难以兼容外部生态中已存在的大量 Skill 包结构。若不先把 Skill 运行时、安装流程和生态兼容能力打牢，后续市场、工作流和团队复用都会建立在脆弱基础之上。需要把 Skill 系统从本地脚本发现机制升级为正式的平台能力。

当前 change 聚焦于Skill 系统升级，需要把 proposal 中的能力边界落到可实现、可验证、可归档的设计基线。

## Goals / Non-Goals

**Goals:**
- 建立渐进式加载策略：区分元数据、文档、脚本和运行依赖的加载阶段，按需展开，减少 token 和 I/O 开销
- 建立完整的安装生命周期：补齐安装、升级、卸载、启用/禁用、依赖检查和失败回滚流程
- 建立生态兼容层：兼容 OpenClaw/NanoClaw 等 Skill 目录结构、frontmatter 规范和脚本组织方式
- 建立 Skill 调试与诊断能力：支持查看输入输出、运行日志、依赖异常和版本来源，便于定位问题
- 建立更完整的管理体验：在 Skills 页面中提供分类浏览、搜索、筛选、状态切换和调试入口

**Non-Goals:**
- 不在本 change 中重写与当前主题无关的既有业务流程。
- 不为未来假设场景预留超出 proposal 范围的实现细节。
- 不绕过现有 capability 分层引入新的旁路实现。

## Decisions

1. 使用独立 capability 与 delta spec 表达 Skill 系统升级 需求边界，避免实现先行导致范围漂移。
2. 涉及持久化时采用增量迁移与向后兼容策略，避免破坏既有数据与会话状态。
3. 所有关键状态变化都通过结构化接口或事件返回，便于前端消费、测试验证和后续治理。
4. 用户可见入口必须与后端能力同步落地，避免出现“底层可用但界面不可达”的割裂体验。
5. 验收以 capability 契约为准，测试、实现和后续归档均围绕同一份规格演进。

## Risks / Trade-offs

- Skill 系统升级 横跨 skills-system、data-layer、web-backend、web-frontend 分层，若边界定义不清，后续实现容易出现职责重叠。
- Skill 系统升级 一旦缺少稳定状态模型或历史追踪，用户侧很难判断当前结果是否可信。
- 若实现阶段直接扩展超出 proposal 范围的功能，Skill 系统升级 的交付周期和验证成本都会明显上升。

## Migration Plan

- 先落地数据模型、迁移脚本和索引调整，再接入上层服务逻辑。
- 随后实现核心运行时或服务层集成，确保 capability 契约先稳定。
- 最后交付 API 与前端入口，确保用户只看到已可用且可验证的能力。
- 完成后执行 OpenSpec 校验与针对该主题的实现级回归验证。

## Open Questions

- 当前无阻塞性开放问题；若实现阶段引入新的外部依赖、权限边界或迁移风险，再回填对应约束。
