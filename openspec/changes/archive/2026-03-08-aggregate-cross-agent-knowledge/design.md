## Context

用户在 OKK 之外还会长期使用 Claude Code、OpenClaw、NanoClaw 等工具，这些工具中已经积累了大量对项目有价值的会话、记忆和约定。如果这些资产无法被 OKK 识别和吸收，用户就不得不在多个工具之间重复维护上下文，长期记忆也会被割裂。需要把“跨工具知识汇聚”定义为正式能力，让 OKK 成为统一的工程知识入口而不是新的孤岛。

当前 change 聚焦于跨 Agent 知识聚合，需要把 proposal 中的能力边界落到可实现、可验证、可归档的设计基线。

## Goals / Non-Goals

**Goals:**
- 建立多来源导入适配层：支持读取 Claude Code 本地会话、`CLAUDE.md` 记忆文件以及后续其他 Agent 工具的数据源
- 建立统一导入契约：把不同来源的消息、记忆和元数据归一化为一致的知识导入格式，便于治理和扩展
- 建立来源与证据链：为每条导入知识记录来源工具、原始文件或会话引用，确保后续可追溯、可审核
- 建立去重与合并流程：识别与现有知识重复、重叠或冲突的内容，并提供合并、覆盖或忽略策略
- 建立导入向导与回顾能力：支持用户分批导入、预览结果、选择保留项，并回看历史导入记录

**Non-Goals:**
- 不在本 change 中重写与当前主题无关的既有业务流程。
- 不为未来假设场景预留超出 proposal 范围的实现细节。
- 不绕过现有 capability 分层引入新的旁路实现。

## Decisions

1. 使用独立 capability 与 delta spec 表达 跨 Agent 知识聚合 需求边界，避免实现先行导致范围漂移。
2. 涉及持久化时采用增量迁移与向后兼容策略，避免破坏既有数据与会话状态。
3. 所有关键状态变化都通过结构化接口或事件返回，便于前端消费、测试验证和后续治理。
4. 用户可见入口必须与后端能力同步落地，避免出现“底层可用但界面不可达”的割裂体验。
5. 验收以 capability 契约为准，测试、实现和后续归档均围绕同一份规格演进。

## Risks / Trade-offs

- 跨 Agent 知识聚合 横跨 knowledge-import、knowledge-engine、data-layer、web-backend、web-frontend 分层，若边界定义不清，后续实现容易出现职责重叠。
- 跨 Agent 知识聚合 一旦缺少稳定状态模型或历史追踪，用户侧很难判断当前结果是否可信。
- 若实现阶段直接扩展超出 proposal 范围的功能，跨 Agent 知识聚合 的交付周期和验证成本都会明显上升。

## Migration Plan

- 先落地数据模型、迁移脚本和索引调整，再接入上层服务逻辑。
- 随后实现核心运行时或服务层集成，确保 capability 契约先稳定。
- 最后交付 API 与前端入口，确保用户只看到已可用且可验证的能力。
- 完成后执行 OpenSpec 校验与针对该主题的实现级回归验证。

## Open Questions
