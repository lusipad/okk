## Context

Skill 生态是合伙人持续成长的关键来源，但“只能扫描本地目录”的模式严重限制了可发现性、可安装性和生态复用。既然外部生态已经沉淀了大量可复用 Skills，OKK 需要一个正式的市场能力来完成发现、安装、更新和反馈闭环，让用户不必手动搬运脚本，也让 Skill 成为可运营的产品能力而不是隐藏功能。

当前 change 聚焦于Skill 市场，需要把 proposal 中的能力边界落到可实现、可验证、可归档的设计基线。

## Goals / Non-Goals

**Goals:**
- 建立远程 Skill 注册表接入：拉取并缓存市场元数据，支持版本、来源、标签和下载类型等字段
- 建立安装与更新闭环：支持一键安装、升级、卸载和失败回滚，保留来源与版本状态
- 建立发现与筛选体验：支持按分类、用途、评分、兼容性和关键字浏览与搜索 Skills
- 建立评价与使用反馈：展示安装量、使用频率、评分或推荐信息，帮助用户判断 Skill 质量
- 建立市场与本地系统联动：已安装 Skill 的状态、来源和可升级信息在市场与本地管理页保持一致

**Non-Goals:**
- 不在本 change 中重写与当前主题无关的既有业务流程。
- 不为未来假设场景预留超出 proposal 范围的实现细节。
- 不绕过现有 capability 分层引入新的旁路实现。

## Decisions

1. 使用独立 capability 与 delta spec 表达 Skill 市场 需求边界，避免实现先行导致范围漂移。
2. 涉及持久化时采用增量迁移与向后兼容策略，避免破坏既有数据与会话状态。
3. 所有关键状态变化都通过结构化接口或事件返回，便于前端消费、测试验证和后续治理。
4. 用户可见入口必须与后端能力同步落地，避免出现“底层可用但界面不可达”的割裂体验。
5. 验收以 capability 契约为准，测试、实现和后续归档均围绕同一份规格演进。

## Risks / Trade-offs

- Skill 市场 横跨 skill-marketplace、skills-system、data-layer、web-backend、web-frontend 分层，若边界定义不清，后续实现容易出现职责重叠。
- Skill 市场 一旦缺少稳定状态模型或历史追踪，用户侧很难判断当前结果是否可信。
- 若实现阶段直接扩展超出 proposal 范围的功能，Skill 市场 的交付周期和验证成本都会明显上升。

## Migration Plan

- 先落地数据模型、迁移脚本和索引调整，再接入上层服务逻辑。
- 随后实现核心运行时或服务层集成，确保 capability 契约先稳定。
- 最后交付 API 与前端入口，确保用户只看到已可用且可验证的能力。
- 完成后执行 OpenSpec 校验与针对该主题的实现级回归验证。

## Open Questions

- 当前无阻塞性开放问题；若实现阶段引入新的外部依赖、权限边界或迁移风险，再回填对应约束。
