## Context

当前 RepositoryContextService 只会为仓库上下文拼接固定数量的已发布知识摘要，无法根据用户当前问题动态挑选相关条目；QA Gateway 也没有把“本次回答使用了哪些知识”作为结构化信息返回给前端。与此同时，知识建议卡只有保存/忽略两种动作，缺少保存前校正内容的能力。这个 change 横跨 core、backend 和 frontend，需要先锁定检索、回传和交互边界。

## Goals / Non-Goals

**Goals:**
- 将固定 top-N 摘要改为查询相关知识注入，并区分背景知识与问题相关知识
- 在 QA Gateway 中回传本次实际使用的知识引用，并对被注入条目更新 `view_count`
- 在 Chat UI 中展示知识引用来源
- 为 `KnowledgeSuggestionCard` 增加保存前编辑标题、内容和标签的能力，并支持保存后跳转

**Non-Goals:**
- 不在本 change 中重写知识条目的 CRUD、治理和版本历史 API
- 不在本 change 中引入新的向量检索、嵌入模型或外部搜索依赖
- 不在本 change 中改造 KnowledgePage 本身的列表/编辑工作台

## Decisions

1. 使用“背景知识 + 查询相关知识”双层注入模型。
   - 背景知识来自仓库级稳定知识，用于维持长期上下文；查询相关知识基于当前用户问题执行 FTS 检索。
   - 备选方案：完全取消背景知识，仅保留查询检索。拒绝原因：对仓库长期约定的稳定注入会退化。
2. 查询相关知识仍以现有 FTS5 和知识元数据为基础，不新增独立检索基础设施。
   - 备选方案：引入向量检索。拒绝原因：proposal 未要求，且当前仓库已有 FTS 能力足以支撑第一阶段。
3. 只有真正进入本次上下文构建结果的知识条目才会更新 `view_count` 并返回给前端。
   - 备选方案：对所有候选检索结果累计 `view_count`。拒绝原因：会污染使用统计，无法准确表达“实际被引用”。
4. QA Gateway 将知识引用作为结构化 payload 返回给 Chat UI，而不是让前端自行推断。
   - 备选方案：前端根据消息文本再做二次匹配。拒绝原因：会造成不一致且难以测试。
5. `KnowledgeSuggestionCard` 在前端维护临时编辑态，保存时将用户修正后的标题、内容和标签一并提交。
   - 备选方案：点击保存后跳转到独立编辑器。拒绝原因：会打断当前对话流，且超出本 change 目标。

## Risks / Trade-offs

- [检索结果噪声过高] → 用背景/相关分层和条目数量上限控制注入规模
- [引用与回答不一致] → 以 QA Gateway 最终生成的上下文快照作为唯一知识引用来源
- [建议卡编辑增加前端状态复杂度] → 仅在卡片局部维护草稿态，不引入全局状态容器

## Migration Plan

- 先重构 RepositoryContextService 的知识选择与上下文装配逻辑
- 再在 QA Gateway 中串联知识检索、使用计数和引用回传
- 最后交付 Chat UI 引用展示与 `KnowledgeSuggestionCard` 编辑交互，并补齐测试
- 如需回滚，可恢复固定摘要策略并隐藏前端知识引用展示，不涉及数据库回退

## Open Questions
