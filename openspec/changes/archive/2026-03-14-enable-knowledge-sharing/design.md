## Context

知识治理、知识导入和基础 CRUD 已经存在，但知识条目仍默认停留在个人作用域内。仓库里已有 `memory-sharing` 的审核流与概览接口实现思路，当前 change 需要把这条模式落到知识域，同时避免把标准文件导入导出和订阅分发混进同一实现。

当前约束有三点：
- 共享必须沿用现有本地优先、可审计的架构，不能引入额外远程权限系统
- 已发布共享知识与个人知识必须共用同一条 KnowledgeEntry 主数据，避免复制两套知识模型
- 审核和团队浏览要能被前端直接消费，并保持与现有 governance / imports 页面一致的工作台风格

## Goals / Non-Goals

**Goals:**
- 为知识条目建立从个人条目到共享发布的完整状态流转
- 建立共享请求、审核、发布和审计记录的后端与持久化契约
- 提供团队知识浏览与审核工作台，让共享知识成为可运营入口
- 保持与现有 knowledge-engine 检索、状态流转和治理流程兼容

**Non-Goals:**
- 不在本 change 中定义 Markdown/YAML 标准格式导入导出
- 不在本 change 中引入知识源订阅、通知或自动同步机制
- 不在本 change 中升级到远程多租户鉴权模型

## Decisions

1. **共享采用独立附属记录，不复制 KnowledgeEntry 主体**
   共享请求、审核结论和发布时间保存在独立 sharing 记录中，KnowledgeEntry 只扩展最少的可见性/共享状态字段。这样可以复用现有版本历史和治理链路，避免一条知识在“个人版/团队版”两份数据之间漂移。

2. **审核流使用显式状态机**
   状态限定为 `draft_request -> pending_review -> approved -> published`，并允许 `rejected` / `changes_requested` 分支。相较于用多个布尔字段拼装，显式状态机更容易做 API 校验、前端展示和审计追踪。

3. **团队浏览只暴露已发布共享条目**
   团队知识库查询不直接暴露待审条目，审核视图与团队浏览视图分离。这样能降低误曝光风险，也让团队列表保持稳定语义。

4. **沿用现有工作台模式组织前后端接口**
   后端提供请求列表、详情、审核动作、团队概览四类接口；前端以共享面板、审核队列、团队列表三块界面承载。这样可与现有 Governance / Imports 页面保持相似心智模型。

5. **共享动作写入完整审计链**
   每次提交、审批、驳回、发布都写入 review/audit 记录，并引用操作者、时间和备注。相比只保留最终状态，这能支持后续治理与回滚说明。

## Risks / Trade-offs

- [共享状态与知识状态耦合过深] → 仅把共享可见性和发布态作为知识扩展字段，审核细节全部落到独立 sharing 记录，降低状态交叉复杂度
- [团队浏览误暴露未审内容] → 团队查询默认只返回 `published` 共享记录，待审内容只能通过审核接口查看
- [前端入口分散导致体验割裂] → 共享入口统一挂在 Knowledge 工作台及治理相关导航下，避免出现平行但不相连的页面
- [后续权限模型变化影响当前设计] → 当前只定义本地团队审核基础契约，不预埋远程 RBAC 细节，后续通过独立 change 扩展

## Migration Plan

1. 在 data-layer 中补齐 sharing 记录、review 记录和必要索引
2. 在 knowledge-engine 上接入共享状态流转、团队查询和审计写入
3. 暴露共享请求、审核、团队列表与概览 API
4. 交付前端共享面板、审核工作台和团队知识浏览界面
5. 补齐共享流转、可见性过滤和审计场景测试

## Open Questions

- 当前无阻塞性开放问题；如果后续需要更细粒度的团队角色模型，再通过独立 change 扩展
