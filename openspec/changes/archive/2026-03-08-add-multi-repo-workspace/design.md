## Context

真实工程工作经常横跨多个仓库：前端、后端、SDK、脚本仓、文档仓往往需要联动理解和同时修改。当前 OKK 以单仓库为中心组织会话和上下文，用户一旦切换项目就容易丢失历史选择、仓库关系和跨仓库记忆，导致多项目任务只能靠人工补全背景。需要引入 workspace 级抽象，把“多个仓库组成一个工作空间”升级为正式的一等能力。

当前 change 聚焦于多仓库工作空间，需要把 proposal 中的能力边界落到可实现、可验证、可归档的设计基线。

## Goals / Non-Goals

**Goals:**
- 建立多仓库 workspace 模型：允许用户把多个仓库组织到一个工作空间下，并维护名称、描述、默认仓库和活跃仓库
- 建立 workspace 级上下文：支持在同一会话中携带多个仓库的背景信息、已发布知识和关联记忆，减少频繁切换造成的上下文丢失
- 支持仓库切换与附加目录协作：在工作台里快速切换当前主仓库，并在需要时将其他仓库作为附加上下文参与任务执行
- 支持跨仓库检索与导航：对 workspace 内代码、知识和会话进行统一搜索，并能跳转回具体仓库和文件位置
- 持久化 workspace 配置：保存用户的仓库分组、默认排序、最近活跃范围和常用组合，支持继续上次工作

**Non-Goals:**
- 不在本 change 中重写与当前主题无关的既有业务流程。
- 不为未来假设场景预留超出 proposal 范围的实现细节。
- 不绕过现有 capability 分层引入新的旁路实现。

## Decisions

1. 使用独立 capability 与 delta spec 表达 多仓库工作空间 需求边界，避免实现先行导致范围漂移。
2. 涉及持久化时采用增量迁移与向后兼容策略，避免破坏既有数据与会话状态。
3. 所有关键状态变化都通过结构化接口或事件返回，便于前端消费、测试验证和后续治理。
4. 用户可见入口必须与后端能力同步落地，避免出现“底层可用但界面不可达”的割裂体验。
5. 验收以 capability 契约为准，测试、实现和后续归档均围绕同一份规格演进。

## Risks / Trade-offs

- 多仓库工作空间 横跨 workspace-management、repo-management、knowledge-engine、data-layer、web-backend、web-frontend 分层，若边界定义不清，后续实现容易出现职责重叠。
- 多仓库工作空间 一旦缺少稳定状态模型或历史追踪，用户侧很难判断当前结果是否可信。
- 若实现阶段直接扩展超出 proposal 范围的功能，多仓库工作空间 的交付周期和验证成本都会明显上升。

## Migration Plan

- 先落地数据模型、迁移脚本和索引调整，再接入上层服务逻辑。
- 随后实现核心运行时或服务层集成，确保 capability 契约先稳定。
- 最后交付 API 与前端入口，确保用户只看到已可用且可验证的能力。
- 完成后执行 OpenSpec 校验与针对该主题的实现级回归验证。

## Open Questions

- 当前无阻塞性开放问题；若实现阶段引入新的外部依赖、权限边界或迁移风险，再回填对应约束。
