## Context

仓库已经有知识导入批次、导入预览和确认机制，但它们主要服务于跨 Agent 数据吸收，缺少一套稳定的知识文件交换格式。当前 change 需要在不新建额外根 capability 的前提下，为知识条目补齐标准化导出/导入协议，并尽量复用现有导入确认链路。

当前约束有三点：
- 根 spec 中不存在 `knowledge-import`，因此不能把它作为 modified capability 依赖
- 标准格式必须覆盖知识条目的核心字段，同时避免把内部实现细节泄露到文件协议
- 标准文件导入需要与现有导入预览/确认体验保持兼容，而不是另起一套流程

## Goals / Non-Goals

**Goals:**
- 定义 Markdown + YAML frontmatter 的知识文件协议
- 提供单条和批量知识导出能力，支持可读可编辑的离线文件
- 提供标准文件导入预览、校验和确认流程
- 为协议引入显式格式版本，保证后续字段演进可兼容

**Non-Goals:**
- 不在本 change 中处理团队共享审批和发布
- 不在本 change 中实现知识源订阅、同步通知或自动拉取
- 不在本 change 中支持多种外部平台导出格式

## Decisions

1. **协议以 Markdown 正文 + YAML frontmatter 组织**
   正文保留知识主体内容，frontmatter 承载标题、标签、分类、状态、来源、格式版本等结构化元数据。这样既便于人类编辑，也便于脚本解析。

2. **格式版本采用显式字段**
   每个导出文件必须包含 `format_version`。相比隐式兼容，显式版本更有利于后续字段扩展和导入时给出兼容性提示。

3. **导入走现有预览/确认链路**
   标准文件导入不新建独立审批模型，而是复用现有 import preview / confirm 流程，只在输入解析和校验阶段增加 frontmatter 处理。这样能减少重复实现。

4. **导出内容只包含稳定领域字段**
   文件协议聚焦 title、summary、content、category、tags、status、source refs 等稳定字段，不暴露 DAO 内部 ID 或仅对运行时有意义的瞬时字段。

5. **批量导出采用目录清单而非单一大文件**
   每条知识导出为独立 Markdown 文件，批量导出时额外生成清单索引。相比拼成单文件，独立文件更利于版本控制、手动审阅和选择性导入。

## Risks / Trade-offs

- [文件协议字段过多导致兼容成本上升] → 协议只覆盖稳定字段，新增字段必须通过 `format_version` 受控演进
- [标准导入与现有跨 Agent 导入流程分叉] → 明确复用现有预览/确认链路，只扩展解析器和校验器
- [用户手工编辑 frontmatter 导致格式错误] → 导入阶段提供结构化校验错误并阻止不合法文件进入确认环节
- [批量导出文件过多影响管理] → 通过批量清单文件和约定命名减轻目录管理成本

## Migration Plan

1. 先定义 frontmatter 字段集合、版本字段和文件命名约定
2. 在 knowledge-engine 中实现导出序列化与标准文件解析器
3. 在 web-backend 中接入导出接口和标准文件导入预览接口
4. 在 web-frontend 中交付导出入口、文件导入和校验反馈界面
5. 补齐格式兼容、批量导出和错误输入场景测试

## Open Questions

- 当前无阻塞性开放问题；若后续需要支持其他知识文件协议，再通过独立 change 扩展
