## Context

单个 Skill 更像一个能力原子，而真实工程工作往往需要一组固定顺序、带条件判断和上下文传递的步骤。缺少工作流层时，用户只能靠记忆手动组织“先做什么、再做什么、失败后怎么办”，这会让高频流程无法复用，也让 Agent 与 Skill 的组合能力难以沉淀。需要把“Skill 组合成工作流”升级为正式能力，让 OKK 可以复用工程 SOP 而不仅是单次执行命令。

当前 change 聚焦于Skill 工作流，需要把 proposal 中的能力边界落到可实现、可验证、可归档的设计基线。

## Goals / Non-Goals

**Goals:**
- 建立工作流定义格式：支持描述节点、执行顺序、输入输出映射、条件分支、失败处理和人工确认点
- 建立工作流运行模型：把 Agent、Skill、提示模板和上下文注入组织成可复用节点，并在执行中传递状态
- 建立模板库与分享机制：预置常见工程工作流模板，并支持保存团队自定义流程供重复使用
- 建立运行与观察能力：记录每次工作流执行结果、步骤状态和失败原因，便于重试与复盘
- 建立编辑与管理入口：支持在界面中创建、编辑、复制和发布工作流模板

**Non-Goals:**
- 不在本 change 中重写与当前主题无关的既有业务流程。
- 不为未来假设场景预留超出 proposal 范围的实现细节。
- 不绕过现有 capability 分层引入新的旁路实现。

## Decisions

1. 使用独立 capability 与 delta spec 表达 Skill 工作流 需求边界，避免实现先行导致范围漂移。
2. 优先复用现有分层，不为 Skill 工作流 引入不必要的新基础设施。
3. 所有关键状态变化都通过结构化接口或事件返回，便于前端消费、测试验证和后续治理。
4. 用户可见入口必须与后端能力同步落地，避免出现“底层可用但界面不可达”的割裂体验。
5. 验收以 capability 契约为准，测试、实现和后续归档均围绕同一份规格演进。

## Risks / Trade-offs

- Skill 工作流 横跨 skill-workflows、skills-system、agent-system、web-backend、web-frontend 分层，若边界定义不清，后续实现容易出现职责重叠。
- Skill 工作流 一旦缺少稳定状态模型或历史追踪，用户侧很难判断当前结果是否可信。
- 若实现阶段直接扩展超出 proposal 范围的功能，Skill 工作流 的交付周期和验证成本都会明显上升。

## Migration Plan

- 随后实现核心运行时或服务层集成，确保 capability 契约先稳定。
- 最后交付 API 与前端入口，确保用户只看到已可用且可验证的能力。
- 完成后执行 OpenSpec 校验与针对该主题的实现级回归验证。

## Open Questions
