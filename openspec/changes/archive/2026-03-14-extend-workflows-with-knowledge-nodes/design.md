## Context

当前工作流能力已经支持 `prompt`、`skill`、`agent`、`condition` 四类节点，并通过 `workflowsRoutes` 与 `WorkflowsPage` 提供基础 CRUD、运行与历史查看能力。但工作流仍无法直接消费知识库内容，导致用户只能把知识复制到 prompt 模板里手工传递，既不稳定，也无法在运行历史中追踪到底引用了哪些知识条目。

这个 change 聚焦于把知识作为工作流的一等输入节点，而不是结果产物。设计需要在不重写现有工作流持久化结构的前提下，把知识引用补入现有节点模型、运行时上下文与前端编辑器。

## Goals / Non-Goals

**Goals:**
- 为工作流新增 `knowledge_ref` 节点类型，并保持与既有节点序列化结构兼容。
- 支持在节点配置中声明知识来源、查询约束和输出键，让后续节点能够直接消费结构化知识上下文。
- 在后端运行时解析知识节点，把知识条目加载为统一的上下文对象，并记录到运行步骤输出。
- 在前端编辑器和运行详情中提供知识节点的创建、配置、预览和运行结果可见性。

**Non-Goals:**
- 不在本 change 中实现工作流结果保存为知识；该能力由 `capture-workflow-results-as-knowledge` 单独负责。
- 不在本 change 中重写整个工作流编辑器为图形化画布。
- 不在本 change 中引入新的工作流持久化表结构；节点仍然复用现有 `nodes_json`。

## Decisions

1. **新增节点类型而不是扩展 prompt 语义**
   - 选择：在 `SkillWorkflowNodeType` 中新增 `knowledge_ref`。
   - 原因：把知识输入建模为独立节点，能让运行历史、校验逻辑和前端配置都保持显式，不把知识拼装隐藏在 prompt 模板里。
   - 备选：继续使用 `prompt` 节点拼接知识文本。
   - 放弃原因：无法追踪知识来源，也无法在 API 校验和运行步骤中识别知识输入。

2. **节点配置使用“显式条目 + 可选筛选”最小模型**
   - 选择：`knowledge_ref` 节点配置至少包含 `outputKey`，并允许两类来源：
     - `entryIds: string[]`，用于显式引用固定知识条目；
     - `filters`，包含 `repoId`、`category`、`tags`、`status`、`query`、`limit` 等约束，用于运行时搜索。
   - 原因：同时覆盖“人工挑选知识”和“按主题动态取知识”两类场景，又不需要新增独立节点子类型。
   - 备选：只允许固定 `entryIds`。
   - 放弃原因：无法支持面向仓库约定或治理规则的动态装配场景。

3. **运行时输出结构统一为结构化知识包**
   - 选择：知识节点执行后在 `context[outputKey]` 中写入 `{ entries, summary }`，其中 `entries` 为可序列化的知识摘要数组，`summary` 为供 prompt/agent 直接消费的拼接文本。
   - 原因：后续节点既可以消费文本，也可以消费结构化条目元数据；运行步骤也能稳定展示解析结果。
   - 备选：只输出一段长文本。
   - 放弃原因：会丢失条目 ID、分类、标签等结构化信息，不利于调试和后续扩展。

4. **后端负责节点校验与解析，前端只负责配置与预览**
   - 选择：`web-backend` 在工作流创建、更新、运行时校验 `knowledge_ref` 的必要字段和查询约束；`web-frontend` 负责表单输入、节点 JSON 展示和运行结果展示。
   - 原因：避免前端和后端各自维护一套不一致的节点合法性规则。
   - 备选：仅前端校验。
   - 放弃原因：直接写 API 或导入模板时无法得到统一约束。

## Risks / Trade-offs

- [知识查询结果过大] → 通过 `limit` 和 `summary` 截断策略控制运行时上下文大小，并在节点配置中要求明确输出键。
- [动态筛选带来非确定性] → 在运行步骤输出中记录实际命中的知识条目 ID、标题和查询条件，保证运行可回溯。
- [前端暂时仍以 JSON 编辑为主] → 先补齐节点类型、字段说明和结果预览，不在本 change 中引入复杂可视化编辑器。

## Migration Plan

- 先扩展核心类型与运行时解析逻辑，确保 `knowledge_ref` 节点能被持久化和执行。
- 再补齐工作流 API 的节点校验、模板序列化和运行步骤输出。
- 最后交付前端配置入口、节点预览和运行详情展示，并补充测试。

## Open Questions

- 当前不引入权限过滤模型，默认沿用现有知识查询的可见范围。
- 如后续需要支持“按相似度推荐知识”而非显式筛选，可在此节点类型上继续扩展 `filters`，不新增节点类别。
