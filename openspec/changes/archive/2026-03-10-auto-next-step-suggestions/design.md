## Context

一轮回答结束后，用户经常需要停下来思考下一句该怎么问，导致对话主流程在最关键的时刻重新回到空白输入框。当前系统已经拥有足够多的上下文信息，但还没有把“下一步”显式表达出来。V1 目标不是让模型继续替用户思考，而是用规则驱动的低成本方式减少续写摩擦。

这个 change 需要在 ChatPage 中基于最后消息内容与当前上下文生成 2 到 3 个建议按钮，并把点击行为限制为回填 Composer，而不是自动发送或新增 LLM 请求。

## Goals / Non-Goals

**Goals:**
- 在会话主舞台中展示规则驱动的下一步建议
- 基于最后一条消息与当前上下文生成 2 到 3 个建议动作
- 建议点击后仅注入 Composer，由用户手动决定是否发送
- 保持规则实现可预测、可验证、可扩展

**Non-Goals:**
- 不发起额外的 LLM 调用生成建议
- 不将建议结果持久化到会话或数据库
- 不在本 change 中实现用户级个性化排序或学习机制
- 不把建议系统扩展成完整工作流编排器

## Decisions

1. 建议生成采用纯前端规则函数，输入限定为最后一条消息、会话状态和已有上下文，避免引入新的网络往返或模型成本。
2. 建议只在上一轮回答完成且当前不处于流式输出时显示，避免在执行中出现误导性的下一步按钮。
3. 建议数量上限为 3，避免聊天底部再次形成高密度按钮墙。
4. 点击建议后只向 Composer 注入可编辑草稿，不自动发送，保持用户最终控制权。
5. 当规则无法给出高置信建议时，直接不展示建议区，优先保持界面克制。

## Risks / Trade-offs

- [规则命中率不足] -> 允许无命中时不展示建议，而不是硬凑低质量建议。
- [建议与真实上下文脱节] -> 规则只读取当前会话已知状态，不引入猜测性的远端数据。
- [按钮过多影响阅读] -> 严格限制建议数量，并仅在回答完成后显示。

## Migration Plan

- 先在 ChatPage 中实现建议规则函数与 view model。
- 再新增 NextStepSuggestions 组件并接入 Composer 草稿注入。
- 最后补齐流式中隐藏、无命中隐藏和点击注入的验收验证。

## Open Questions

- 当前无阻塞性开放问题；如后续需要更复杂的建议来源，再通过新的 capability 扩展规则或引入模型辅助。
