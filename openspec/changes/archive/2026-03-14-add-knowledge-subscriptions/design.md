## Context

团队共享知识解决了“能发布”的问题，但没有解决“如何持续消费更新”的问题。当前仓库没有订阅关系、同步游标和更新通知契约，因此用户只能被动浏览团队知识库。这个 change 需要为知识源订阅建立独立模型，并保持它与共享发布、个人导入和工作台管理之间的边界清晰。

当前约束有三点：
- 订阅不是共享审批的延伸动作，而是面向消费侧的长期关系模型
- 订阅内容导入必须复用现有知识创建/导入能力，避免维护第二套落库流程
- 订阅需要持久化同步状态，否则无法支持“最近更新”和增量消费

## Goals / Non-Goals

**Goals:**
- 为团队、项目或主题知识源建立订阅关系模型
- 为订阅维护最近同步时间、游标和待消费更新记录
- 提供订阅更新列表和一键导入入口
- 让前端具备订阅管理、查看更新和导入订阅知识的工作台能力

**Non-Goals:**
- 不在本 change 中实现共享审批、共享发布或团队治理看板
- 不在本 change 中实现自动后台推送或实时长连接通知
- 不在本 change 中引入跨组织或远程多租户同步模型

## Decisions

1. **订阅关系与共享记录分层建模**
   共享记录表达“哪些知识可以被团队看到”，订阅关系表达“哪个用户持续关注哪些来源”。两者分层后，审核与消费不互相污染。

2. **更新发现采用拉取式增量游标**
   每条订阅维护最近同步时间或变更游标，后端按游标拉取新增/变更共享知识。相比实时推送，拉取式方案更贴合当前本地优先架构，也更容易测试。

3. **订阅更新与个人知识导入解耦**
   更新列表只表达“可消费内容”，真正落到个人知识库时再调用现有知识创建/导入逻辑。这样可以避免订阅系统直接操纵知识主表。

4. **订阅源统一抽象为 source descriptor**
   无论来源是团队、项目还是主题，后端都使用统一 source descriptor 存储 source type、source id 和显示元数据，减少 API 与前端分支判断。

5. **更新通知先做工作台列表，不做实时推送**
   当前 change 只要求用户进入工作台后能看到订阅更新和未消费项，不引入 websocket/push。这样能把复杂度控制在当前明确需求内。

## Risks / Trade-offs

- [订阅与共享边界混乱] → 订阅只面向已发布共享知识，任何待审状态都不得进入订阅更新流
- [更新记录无限增长] → 通过同步游标与消费状态控制查询范围，必要时后续再补归档策略
- [一键导入导致重复知识] → 导入仍走既有知识创建/导入校验逻辑，沿用去重与确认约束
- [未来需要实时通知] → 当前先固定为拉取式工作台列表，后续通过独立 change 增加推送通道

## Migration Plan

1. 在 data-layer 中新增订阅关系、同步状态和更新记录模型
2. 在 knowledge-engine 中实现来源跟踪、增量更新查询和导入映射
3. 在 web-backend 中暴露订阅 CRUD、更新列表和一键导入接口
4. 在 web-frontend 中交付订阅管理、更新列表和导入交互
5. 补齐订阅同步、更新可见性和重复导入场景测试
