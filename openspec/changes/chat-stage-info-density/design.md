## Context

Chat 主舞台已经承载主要工作流，但头部区目前缺少足够的轻量元信息。用户想知道当前会话规模、启用了多少 Skills 和 MCP、当前工作是否处于轻装还是带能力状态，往往需要切到侧栏或依赖记忆。问题不是没有数据，而是这些状态没有在用户最常注视的位置以低噪音形式表达出来。

这个 change 的目标是在不增加后端依赖、不引入额外轮询的前提下，用现有前端状态为 chat-stage-header 增加紧凑元信息，让用户能快速扫读当前会话状态。

## Goals / Non-Goals

**Goals:**
- 在 chat-stage-header 中展示消息数、活跃 Skills 数和活跃 MCP 数等轻量元信息
- 所有元信息均从现有会话与能力状态中本地派生
- 保持头部视觉克制，不抢占消息阅读流
- 让元信息随着会话与能力选择即时更新

**Non-Goals:**
- 不新增任何后端接口、轮询或遥测采集
- 不在本 change 中加入耗时、token 或成本等复杂运行指标
- 不把 chat-stage-header 改造成第二个控制面板
- 不改变首页摘要卡、顶栏上下文或侧栏导航职责

## Decisions

1. 元信息全部从 ChatPage 已有状态本地推导，包括 messages.length、selectedSkillIds 和 selectedMcpServerIds，避免新增数据依赖。
2. 元信息以紧凑标签或短文本呈现，强调可扫读而不是可操作，避免与顶栏操作区职责冲突。
3. 当某项指标为 0 或当前为空会话时，仍保持稳定布局，可展示中性值或低强调文案，而不是让头部结构跳变。
4. 元信息更新遵循现有 React 状态变化，不引入新的 effect 链或额外缓存。

## Risks / Trade-offs

- [头部信息增多影响阅读] -> 限制为少量元信息并使用弱化样式。
- [空会话与有消息会话头部不一致] -> 使用统一元信息容器，避免结构频繁跳变。
- [指标定义不稳定] -> 仅选择当前已有状态可直接计算的字段，不引入解释成本高的新指标。

## Migration Plan

- 先确定 chat-stage-header 的元信息集合与展示顺序。
- 再在 ChatPage 中实现本地派生逻辑并接入头部渲染。
- 最后调整样式与验收用例，验证空会话、长会话和能力开关变化的联动表现。
