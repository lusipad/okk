## Context

当前顶栏已经承担品牌、命令面板和协作面板开关等全局动作，但顶部中央仍缺少稳定的页面定位信息。随着 Chat、Identity、Memory、Workspaces 等页面并列存在，用户切换后很容易只看到一个通用顶栏，而感受不到“我现在在哪个页面、是以谁的身份工作”。

这个 change 需要在不增加复杂路由推断与视觉噪音的前提下，为 ShellLayout 增加轻量的上下文面包屑表达，让页面名和活跃身份成为顶栏的稳定语义。

## Goals / Non-Goals

**Goals:**
- 在顶栏中心展示当前页面名称
- 在可用时附加活跃身份名称，强化合伙人语义
- 允许各页面显式传入自己的顶栏上下文，而不是依赖脆弱的路由字符串推断
- 保持桌面与移动端都具备稳定、低噪音的上下文表达

**Non-Goals:**
- 不实现多级站点地图式 breadcrumb
- 不引入新的全局状态框架或路由中间层
- 不把顶栏改造成高密度信息面板
- 不在本 change 中调整命令面板与协作面板的交互模式

## Decisions

1. ShellLayout 接收页面级 topbar context，而不是自己从 pathname 猜测业务语义。页面自己最清楚当前标题和是否需要附带身份信息。
2. 顶栏上下文保持轻量：页面名是主信息，活跃身份名是可选补充，不在顶部展示冗长说明文案。
3. 当视口变窄时，优先保证页面名可读；身份名可压缩或弱化，但不得让当前页面定位消失。
4. 顶栏上下文只负责表达定位，不承担操作入口，避免与 chat-stage-header 或首页摘要区职责重叠。

## Risks / Trade-offs

- [上下文来源分散在各页面] -> 使用统一的 ShellLayout props 结构，减少不同页面各写一套标题逻辑。
- [顶栏信息过密] -> 控制为页面名加可选身份名，不叠加更多状态指标。
- [移动端空间紧张] -> 优先保留页面名，身份信息在窄屏下可以弱化或缩短展示。

## Migration Plan

- 先为 ShellLayout 增加 topbar context 数据结构与渲染位。
- 再由 Chat、Identity、Memory、Workspaces 等页面显式传入页面上下文。
- 最后补齐桌面与移动端样式和页面切换验证。

## Open Questions

- 当前无阻塞性开放问题；若后续需要更复杂的层级面包屑，再在新的 change 中独立建模。
