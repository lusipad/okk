## Context

整体架构要求 OKK 不只是“能对话”，还要把对话持续沉淀为团队知识，同时保持与真实仓库上下文相连。当前实现虽已具备知识 CRUD、搜索和建议链路，但它们还没有成为用户稳定可理解的工作流。

## Goals / Non-Goals

**Goals:**
- 让仓库上下文成为工作台始终可见、可切换、可恢复的核心状态
- 让知识建议拥有清晰的审核与发布路径
- 让知识条目具备可追溯来源和过时治理能力
- 让知识检索、策展和回流对话构成闭环

**Non-Goals:**
- 不在本 change 中重写 Agent / Skill / MCP 的执行模型
- 不负责桌面壳层诊断和发布流程
- 不扩展新的视觉 token 体系

## Decisions

1. 仓库上下文是一级工作状态，切换仓库必须同步影响当前会话与知识范围。
2. 知识建议不直接混入消息正文，而进入明确的审核/忽略队列。
3. 每条知识必须至少能追踪到来源仓库和会话，优先补充文件、提交和消息级来源。
4. 过时治理基于“发现问题 -> 标记 -> 人工确认 -> 更新或归档”的节奏，不做完全自动覆盖。
5. 知识检索视图要支持 repo、状态、标签和更新时间等多维过滤。

## Risks / Trade-offs

- 如果来源追踪字段设计过重，前后端和数据层改动会明显增大。
- 若仓库切换缺少显式反馈，用户容易误以为仍在旧上下文中工作。
- 过时检测策略若过于激进，会导致大量低质量提醒，削弱知识治理信任。
